Builder keeps the runtime in charge.

Builder is a local-first coding agent CLI where the runtime, not the model, owns routing, tool policy, verification, delegated-run lineage, and completion. PostgreSQL 18 + pgvector backs the operational state, retrieval, and memory layers.

Why Builder feels different

The runtime treats model output as a candidate, not the final authority. That changes how routing, tools, verification, and delegated work are handled.

01 runtime-owned completion

Completion is adjudicated

Builder tracks objectives, evidence, and verification runs before a task is allowed to close. A fluent answer is not enough by itself.

02 tool policy

Tool use is policy-gated

Request routing, execution strategy, and workflow phase determine what can run. Shell, writes, and network access are runtime decisions, not model improvisation.

03 delegated lineage

Delegation stays traceable

Child runs keep parent linkage, dynamic overlays, spawned objectives, and packet records so sub-agent work does not disappear into raw transcript text.

04 retrieval + memory

State stays close to the runtime

Retrieval, workspace indexing, and conversation memory stay attached to the runtime state model instead of being treated as detached prompt glue.

Execution contract

Builder's runtime path is intentionally opinionated: classify, route, execute under policy, then verify and adjudicate before closing the task.

1

Classify

Intent classification and request routing decide whether the turn is planning, investigation, implementation, review, or general execution.

2

Route + constrain

Execution strategy and dynamic profile shape which tools are allowed, which phase is active, and whether delegation should carry an overlay.

3

Record work

Objectives, state cards, workflow steps, recovery routes, and delegated lineage are all written into operational storage as the run progresses.

4

Verify before done

The runtime requires evidence and verification to close the loop instead of trusting a polished final answer on its own.

Quick start

Choose the path that matches how you want to run Builder: local Compose, direct binary download, or the published Docker image.

compose

Run with PostgreSQL + Builder

Good default when you want the full public runtime stack in one command.

curl -O https://raw.githubusercontent.com/agnusdei1207/builder/main/compose.yaml
docker compose up -d

BUILDER_PROJECT_DIR=$(pwd) docker compose run builder
binary

Download the CLI directly

Use GitHub Releases when you want a single binary without the Compose stack.

curl -LO https://github.com/agnusdei1207/builder/releases/latest/download/builder-x86_64-unknown-linux-musl
chmod +x builder-x86_64-unknown-linux-musl
./builder-x86_64-unknown-linux-musl --help
docker image

Use the published image

Handy when you want Builder in a container without copying the public compose file.

docker pull agnusdei1207/builder:latest
docker run --rm -it -v "$(pwd):/workspace" -w /workspace agnusdei1207/builder:latest
